Evaluation Guidelines

This document defines the rules used to score model responses against ground truth in the Venice 39 data-extraction experiment. It is not shown to models during testing — it records the answer-key logic and acceptable equivalences applied after model output is collected. The implementation lives in evaluate.py.

1. Pre-processing

Applied to both ground truth and model values before any comparison:

  1. Lowercase the entire string.
  2. Collapse whitespace (any run of whitespace → single space) and trim leading/trailing spaces.
  3. Apply the abbreviation equivalence dictionary (Section 5) token-by-token, so q.m., q.m, qm, q(uonda)m, and q(uondam) all collapse to quondam before scoring.

For name matching only, an additional punctuation strip ([^\w\s?]) is applied — see Section 2. The ? character is preserved everywhere because it marks illegible glyphs in ground truth.

2. Name matching (which model rows pair with which GT rows)

Each model row (one traveler record) is paired with at most one ground-truth row using greedy best-match on a name-similarity score in [0, 1]:

  • Name normalization: lowercase, strip punctuation, collapse whitespace. ? in either name is treated as a wildcard for one character.
  • Similarity = max of two signals:
    • difflib.SequenceMatcher character-level ratio
    • Token-set overlap: |tokens_a ∩ tokens_b| / max(|tokens_a|, |tokens_b|)
    The token-set component boosts matches where the model reorders or drops minor tokens.
  • Threshold: pairs below similarity 0.70 are not accepted as matches.
  • Tiebreaker: when the same name appears in GT twice (e.g. a traveler with both an arrival and a departure row), arrival- and departure-date alignment are used as secondary signals — preferring the GT row whose dates match the model row.

The result is split into three buckets:

  • Matched: the (model row, GT row) pairs above threshold
  • Unmatched model rows: false positives — counted against precision
  • Unmatched GT rows: false negatives — counted against recall

See evaluate.py:171-224.

3. Precision, recall, F1 per image

  • precision = matched / n_model — of the rows the model produced, how many landed on a real GT traveler
  • recall = matched / n_gt — of the GT travelers, how many the model found
  • F1 = 2 · precision · recall / (precision + recall)

F1 measures traveler identification only — whether the right set of travelers was produced, not whether their fields were transcribed correctly. Per-field accuracy is reported separately (Section 4).

4. Per-field scoring (only on matched rows)

Once a model row has been paired with a GT row, the nine fields are scored independently. A model that fails to find a GT traveler does not get a free pass on the field accuracy of that row — they are accounted for by the lower recall in Section 3.

4a. Text fields (character accuracy)

The fields title, arrival_date, departure_date, place_of_stay, and place_of_origin are scored by character accuracy:

char_accuracy(gt, model) = max(0, 1 − levenshtein(gt, model) / len(gt))

The strings are normalized per Section 1 before measuring edit distance, so a 1-character misspelling on a 16-character GT scores ~0.94 rather than 0, and abbreviation variants count as identical. See evaluate.py:125-145.

4b. Exact-match fields

The fields with_spouse, with_siblings, and children_count are scored by strict equality (1.0 or 0.0). The first two are Y/N booleans; children_count is an integer — they're grouped because all three are scored by exact match rather than by character similarity.

4c. Unscorable cells

If the GT value for a field is empty (e.g. the GT title is blank because the document mentions no title for that traveler), the cell is excluded from both numerator and denominator of the per-field average. It neither helps nor hurts the model.

5. Abbreviation equivalence dictionary

Before any text comparison, each token is replaced by its canonical form using the table below. The mapping is applied identically to GT and model output, so two strings that look different but spell the same Venetian abbreviation score as identical.

The dictionary is intentionally narrow — only abbreviations that GT and models use interchangeably across the same document. Source: evaluate.py:58-75. The same table is also used in the question-answering experiment so per-field numbers are comparable across the three projects.

6. Aggregation across pages, runs, and models

The F1 numbers reported in Section 1 of the summary are micro-averaged within a (model, phase) cell: total_matched / total_model and total_matched / total_gt are summed across every successful image and run, then F1 is computed from those pooled rates. Pages with more travelers (page 36 with 37 travelers vs. page 39 with 10) therefore carry more weight in the aggregate.

Failed runs (parse errors, MAX_TOKENS truncation, etc.) are excluded — neither n_gt nor n_model for a failed image contributes. See evaluate.py:314-319.

7. What evaluation does and does not measure

  • A perfect F1 means the model emitted the right set of named travelers — no more, no less. It says nothing about how accurately the fields for each traveler were transcribed.
  • A perfect place_of_stay accuracy on matched rows means the surviving rows transcribed that field correctly. It says nothing about rows the model missed entirely.
  • The two together — F1 + per-field accuracy on matched rows — are the full picture. A model with F1 = 1.00 and arrival-date accuracy of 0.70 has correctly identified who is on the page but not when they were there.
