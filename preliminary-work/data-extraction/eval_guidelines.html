Evaluation Guidelines — Data Extraction

This document defines how model responses are scored against ground truth in the
Venice 39 data-extraction experiment (per-traveler tabular extraction). It is
not shown to models during collection — it records the answer-key logic
applied afterward. The implementation is evaluate.py; this page is
generated from EVAL_GUIDELINES.md so it cannot silently drift from the code.

This is a different task and scoring scheme from the question-answering
experiment (venice-39-guided / -naive), which scores 22 fixed questions. The two
eval_guidelines pages do not cross-apply. The shared parts are the character-
accuracy method and the expand-both-sides equivalence philosophy.

1. Two-stage scoring

Each model response is a list of traveler records (9 fields each). Scoring has two
stages:

1. Traveler identification — match model rows to ground-truth (GT) rows by
name, then compute precision / recall / F1. This is the headline metric.
2. Per-field accuracy — for each matched row, score each of the other
fields. Reported separately (per field), computed only on matched rows.

F1 measures who is on the page; per-field accuracy measures what the page says
about them. They are independent: a model can score F1 = 1.00 while its
place_of_stay accuracy is 0.80.

2. Row matching (traveler identification)

Each model row is greedily paired with at most one GT row by name similarity:

  • Names are normalized: lowercased, punctuation stripped, whitespace collapsed. A

? (illegible-glyph marker) is kept.

  • Similarity = the larger of (a) difflib.SequenceMatcher character ratio and

(b) token-set overlap |A∩B| / max(|A|,|B|).

  • A pair below 0.70 similarity is not matched.
  • When the same name appears twice in the GT (e.g. one arrival and one departure

in the same week), arrival/departure-date agreement is used as a tiebreaker.

name is used only as the matching key — it is not itself scored as a field.

Precision / recall / F1 (per image):

  • precision = matched / n_model
  • recall = matched / n_gt
  • F1 = 2 · precision · recall / (precision + recall)

Aggregation: F1 is micro-averaged — matched / n_model / n_gt totals are
pooled across all successful images and runs in a (model, phase) cell, then F1 is
computed from the pooled rates. Failed runs are excluded.

3. Per-field scoring (matched rows)

Each field returns a score in [0, 1], or is unscorable (excluded from both
numerator and denominator) when the GT value is empty — a model cannot be
penalized or rewarded for a field the document does not record.

Field(s)Method

place_of_stay, place_of_originCharacter accuracy after place normalization (§5)

Character accuracy = 1 − levenshtein(gt, model) / len(gt), clamped to
[0, 1]. A one-character slip in a 16-character string scores ≈ 0.94, not 0. Both
GT and model are normalized to a canonical form before the edit-distance
comparison, so a correctly-written abbreviation matches the expanded GT exactly.

4. Base normalization (all text fields)

Applied to every text field (title, dates, and — as the first layer — places):

1. Lowercase; collapse whitespace.
2. Strip trailing punctuation ,.;:][ from each token.
3. Strip a trailing ? from GT tokens (illegible marker).
4. Apply the person/general equivalence dictionary (§6), token by token
(expand-both-sides).

This matches venice-39-guided's token preprocessing so the two projects' text
scores are directly comparable.

5. Place normalization (place_of_stay, place_of_origin)

Place fields get base normalization plus Venetian place-name handling, built
from a corpus scan of GT + model outputs:

  • Drop a leading connective in (the document reads "in casa di…"; GT records

"Casa di…", so the in is a connective, not content).

  • Treat commas/colons as separators.
  • Split a glued saint prefix (Sant'Angelo → san angelo).
  • Apply the place equivalence map (§6).

Genuine misreadings of the handwriting (e.g. Feltrina→Pestiora,
Zobenigo→Colenigo) are left intact and scored by character accuracy — the
normalization never rescues them.

6. Equivalence dictionary

Expand-both-sides: GT and model tokens are both normalized to a canonical form
before comparison. Only recognized forms are expanded, so a correct abbreviation
earns full credit while an unrecognized misreading still falls to character
accuracy.

Place-specific (place fields only) — domain decisions by the PI

  • Saint prefix, singular — San / Santo / Santa and all abbreviations

(s. st. sta sta. s.t s.ta s.a st.a sto) collapse to one
canonical san. A bare S. is accepted for any of San/Santo/Santa.

  • Saint prefix, plural — Santi (s.ti ss s.s sti) is kept distinct

as santi. So San ≠ Santi.

  • Cà — Cà / Ca' / Ca / c.a collapse to cà, but not to casa

(Cà [Name] is a distinct building name, not the generic "house of").

  • Lodging abbreviations — loc.→locanda, osp.→ospizio, m.→maria.

Governing principle: accept abbreviations as written on the page and
connective words that appear in the document (in, commas); keep penalizing
genuine misreadings.
