Gemini 3.1 — Named Travelers Across Task Framings

Same model, same three page images, three task framings. The numbers below are mean counts across 10 t = 0 runs. This isolates a behaviour that's easy to miss when only one experiment is in view: Gemini 3.1's recall of who is on the page depends sharply on whether you ask for a count or ask it to enumerate.

2. What the table says

The same model produces two systematically different counts depending on output shape. On page 36, asked directly, Gemini 3.1 says 28 travelers are named. Asked to extract them, it emits a list of 36 traveler records — off by one against the data-extraction GT of 37, but closing nearly the entire eight-traveler shortfall the direct count shows. Structured emission isn't perfect (we'll see a duplication error on page 28 below), but it's dramatically closer than counting in a vacuum.

This isolates task framing as the variable. Enumerating forces the model to "look" once per row; a direct count appears to be pulled from a small-number prior rather than from an exhaustive scan. The gap widens with page size — page 39 (10 GT) is off by 1, page 28 (19 GT) by 1.5–2, page 36 (36 GT) by 7–8.

Guidance helps a little; transcription helps a lot. Adding expert domain guidance nudges the Q&A count up by half a traveler on pages 28 and 36 (17.0 → 17.5 and 28.0 → 28.5), and in the data-extraction setting it closes the page-36 gap by two records (34 → 36). But it doesn't change the model's strategy — the direct count still undershoots, and the page-28 duplication persists across every data-extraction phase.

3. Why the two ground truths differ on page 28

Q13 asks "How many travelers are mentioned by name?" — not "how many unique travelers." Each GT applies a different counting convention, each internally consistent with the task it serves:

  • Q&A GT counts mentions. On page 28, Mandolin Todros appears in the left column as an arrival and again in the right column as a departure (bracketed with Regina Todros). That's two mentions of the same person, so the Q&A GT total is 19.
  • Data-extraction GT counts unique traveler records. The data-extraction schema asks for one row per traveler with separate arrival_date and departure_date fields, so Mandolin Todros contributes one row capturing both events. The DE GT for page 28 is 18.

The extra Gemini 3.1 record on page 28 is not faithful — it is a duplication error

Gemini 3.1 emits 19 records on page 28 in every phase. That number happens to equal the Q&A mention count, but the model isn't answering the Q&A question — it's filling out the data-extraction schema, which asks for one row per traveler. Inspecting the output (see Per-field results — Page 28) confirms: row 1 is "Mandolin Todros" with the arrival date, and the 19th row is a second "Mandolin Todros" instead of being collapsed into row 1's departure_date field. That is the duplication the schema explicitly forbids; it shows up as a 1-row precision dent against the DE GT and produces the F1 = 0.97 plateau we noted in summary.html section 2.
