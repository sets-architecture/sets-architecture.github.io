Venice 39 — Data Extraction Experiment — Summary

3-image dataset, 5 models × 3 phases × 10 runs.

  • N (Naive): image only, minimal prompt (no expert guidance, no transcription).
  • A (Guided): image + expert guidance about Venetian archival conventions.
  • B (Guided + Transcription): image + guidance + full transcription.

How F1 is calculated

F1 in this experiment measures traveler identification — did the model produce the same set of named travelers as the ground truth? Per-field accuracy (date, place of stay, etc.) is reported separately in section 2 and is computed only on rows that successfully matched.

Row matching. Each model output is a list of traveler records. Each record is paired with at most one ground-truth record by greedy best-match on name similarity, computed as the larger of (a) difflib.SequenceMatcher character ratio and (b) token-set overlap, both on a lowercased, punctuation-stripped name (with the quondam abbreviation variants q.m/qm/q(uonda)m/quondam collapsed to one short canonical so they match fully). A ? in either name matches any character (it marks illegible glyphs). Pairs below similarity 0.70 are not matched. When the same name appears twice in the ground truth (e.g. a traveler who arrived and later departed in the same week), arrival- and departure-date alignment is used as a tiebreaker. See evaluate.py:143-178.

Precision / recall / F1 per image.

  • precision = matched / n_model (of the rows the model produced, how many landed on a real GT traveler)
  • recall = matched / n_gt (of the GT travelers, how many the model found)
  • F1 = 2 · precision · recall / (precision + recall)

See evaluate.py:239-247.

Aggregation across images and runs. The F1 numbers in section 1 are micro-averaged: total_matched / total_model and total_matched / total_gt summed across all successful images and runs in a (model, phase) cell, then F1 computed from those pooled rates. Pages with more travelers therefore carry more weight in the aggregate. Failed runs (parse errors, MAX_TOKENS, etc.) are excluded — n_gt / n_model for a failed image are not counted. See evaluate.py:314-319.

What F1 does and does not measure. A perfect F1 means the model emitted the right set of named travelers, no more and no less. It says nothing about whether the dates, place of stay, or family composition for each traveler were transcribed correctly — those are the per-field numbers in section 2. A model can have F1 = 1.00 with a date-field accuracy of 0.70, and that combination is informative: identifying who is on the page is a different skill from transcribing what the page says about them.

2. Per-field accuracy by model × phase

Computed only on rows that matched a GT traveler (so a model that misses half the travelers does not get a free pass on the half it never produced — those are captured by F1 in section 1).

Scoring per field on a matched row:

  • Text fields (title, arrival_date, departure_date, place_of_stay, place_of_origin) — character accuracy = 1 − levenshtein(gt, model) / len(gt) after normalization. A 1-character misspelling on a 16-char string scores ~0.94 rather than 0. Normalization lowercases, collapses whitespace, strips trailing punctuation (and a trailing ? on GT illegible markers), and applies an equivalence dictionary (expand-both-sides, so a correctly-written abbreviation matches the expanded GT).
  • Place fields (place_of_stay, place_of_origin) get extra Venetian place normalization built from a corpus scan: a leading connective in and commas are dropped; the saint prefix San/Santo/Santa (and abbreviations S./St./Sta/S.ta…) collapse to one canonical, while the plural Santi is kept distinct; Cà/Ca'/Ca collapse together but not to Casa; loc.→locanda, osp.→ospizio, m.→maria, and the quondam group (q.m/qm/q(uonda)m…). Genuine misreadings (e.g. Feltrina→Pestiora) are left intact and scored by character accuracy. See evaluate.py norm_place / ABBREVIATION_EXPANSIONS / PLACE_EXPANSIONS.
  • Date fields (arrival_date, departure_date) additionally strip a leading zero from the day (February 09 1769 == February 9 1769; GT is non-padded). Differently-formatted dates (day-first, month abbreviations) still differ — those are deviations from the requested Month D YYYY, not equivalents.
  • Binary fields (with_spouse, with_siblings, children_count) — 1.0 if equal, 0.0 if not.
  • Unscorable cells (e.g. GT title is empty) are excluded from both numerator and denominator — they neither help nor hurt.

This matches venice-39-guided/-naive's text-question scoring so numbers across the three projects are directly comparable.

5. Notes and caveats

  • Gemini 3.1 thinking budget: Gemini 3.1 preview is a thinking model with unbounded default thinking, which caused all 10 Phase A page-36 runs to hit MAX_TOKENS before any JSON was emitted. Resolved by setting thinkingLevel=LOW and max_output_tokens=32768 (collect_responses_gemini31.py:96-99); all current scores reflect successful runs. (thinkingBudget is silently ignored by Gemini 3.1 preview — thinkingLevel is the lever that works.)
  • Claude extended thinking requires temperature=1 (Anthropic API constraint). All other models run at temperature=0.
  • GPT 5.2 needs response_format=json_object enforcement to avoid degenerate-loop generation at temperature=0. The "image" field was also removed from the response schema after empirical evidence that GPT-5.2 hallucinates hex into a literally-named "image" field. See top of collect_responses.py.
  • Claude wraps responses in markdown code fences despite the prompt saying not to. A narrow Claude-only fence-strip is applied before parsing — the only tolerance in the otherwise strict-json.loads pipeline.
  • No ground-truth contamination: collection scripts never read files from gt/. The GT enters the pipeline only at evaluation. Format coincidence (e.g. q.m) comes from the prompt's expert-guidance section, not from the GT.
  • Image normalization: all images are auto-oriented (EXIF rotation baked into pixels, tag stripped) before sending. 18 of the 39 venice-39-guided images had orientation=6 EXIF tags; we verified all current providers auto-orient as of 2026-05-06, but normalization is kept as defensive infrastructure.
